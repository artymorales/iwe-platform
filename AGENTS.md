# AGENTS.md — Инструкции для агента IWE

> Этот файл — главный источник правил для любого агента (Pi Agent, Cline, Claude Code),
> работающего в интеллектуальной рабочей среде (IWE).
>
> **Тип репозитория:** Base (Принципы + Форматы)
> **Иерархия истины:** DS → Pack → Base (при противоречии побеждает вышестоящий)

---

## 1. Архитектура репозиториев

| Репозиторий | Тип | Содержит |
|---|---|---|
| `~/iwe-platform/` | **Base** | Принципы, форматы, протоколы, скрипты, AGENTS.md |
| `~/ds-strategy/` | **DS (governance)** | Цели, WeekPlans, реестры, решения, inbox |
| `~/ds-knowledge-index/` | **DS (knowledge)** | Черновики знаний, captures, drafts |

**Fallback Chain (при противоречии):** DS → Base
**Source-of-truth для знаний:** Pack (когда появится) → DS (пока Pack нет)

## 2. Три стартовых правила

> Мы начинаем с минимально жизнеспособной системы. Остальные правила
> (АрхГейт, мультиагентность, эволюция) добавляются итеративно, по мере
> необходимости.

### Правило 1. ОРЗ-фрактал (Открытие → Работа → Закрытие)

Любая работа проходит три стадии. Пропуск Открытия = незапланированная работа.
Пропуск Закрытия = незафиксированный результат.

| Масштаб | Открытие | Работа | Закрытие |
|---------|----------|--------|----------|
| **Сессия** | Определить задачу, прочитать контекст | Выполнение | Зафиксировать результат |
| **День** | `bash ~/iwe-platform/scripts/day-open.sh` | Между Open и Close | `bash ~/iwe-platform/scripts/day-close.sh` |
| **Неделя** | — | — | `bash ~/iwe-platform/scripts/week-close.sh` |

**Действия агента:**
- При старте сессии — проверить, был ли сделан Day Open сегодня. Если нет — предложить.
- Перед началом новой задачи — явно объявить: что делаем, зачем, какой ожидаемый артефакт.
- По завершению задачи — предложить закрыть (commit + capture, если есть).

### Правило 2. WP Gate (Согласование работы)

Любое новое задание, которого нет в текущем WeekPlan, требует явного согласования
перед началом работы.

**Ритуал согласования:**
1. Агент объявляет: Роль · Работа · Ожидаемый артефакт · Оценка времени.
2. Пользователь даёт явное «да» / «делаем».
3. Только после этого — начало работы.

**Исключения:** исправление ошибки в уже открытой сессии, очевидный багфикс ≤5 мин.

### Правило 3. Capture-to-Pack (Захват знаний)

На каждом рубеже (закрытие сессии/дня/недели) агент проверяет: появилось ли знание,
которое стоит сохранить?

**Типы capture:**
- **Правило** (1-3 строки) → `~/iwe-platform/memory/distinctions.md`
- **Протокол/процесс** → `~/iwe-platform/memory/protocols/`
- **Доменное знание** → `~/ds-knowledge-index/captures/`
- **Урок/рефлексия** → `~/ds-knowledge-index/drafts/`

**Формат capture:**
```markdown
## YYYY-MM-DD: Тема

Что узнал: ...
Откуда: ...
Контекст: ...
```

## 3. Как читать этот файл

- **AGENTS.md** (этот файл) — загружается в начале каждой сессии. Содержит только
  правила верхнего уровня.
- **memory/protocols/** — детальные протоколы, загружаются по триггеру.
- **memory/distinctions.md** — различения, важные для принятия решений.
- **params.yaml** — конфигурация, читаемая скриптами.

## 5. Правила коммуникации

- **Автономность:** не спрашивать разрешения на очевидные действия (сохранение,
  синхронизацию, обновление связанных файлов). Исключения — необратимые действия,
  WP Gate, choice-questions.
- **Ясность:** в начале ответа — краткий план. В конце — резюме сделанного.
- **Ответственность:** если что-то пошло не так — сказать сразу, предложить фикс.
- **Язык:** русский. Технические термины — на английском, если нет устоявшегося
  русского аналога.
