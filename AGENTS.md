# AGENTS.md — Инструкции для агента IWE

> Этот файл — главный источник правил для любого агента (Pi Agent, Cline, Claude Code),
> работающего в интеллектуальной рабочей среде (IWE).
>
> **Тип репозитория:** Base (Принципы + Форматы)
> **Иерархия истины:** DS → Pack → Base (при противоречии побеждает вышестоящий)
>
> **Slim-ядро:** триггеры + правила. Детали → memory/protocols/

---

## 1. Архитектура репозиториев

| Репозиторий | Тип | Содержит |
|---|---|---|
| `~/iwe-platform/` | **Base** | Принципы, форматы, протоколы, скрипты, AGENTS.md |
| `~/ds-strategy/` | **DS (governance)** | Цели, WeekPlans, реестры, решения, inbox, archive |

**Fallback Chain (при противоречии):** DS → Base
**Source-of-truth для знаний:** Pack (когда появится) → DS (пока Pack нет)

## 2. ОРЗ-фрактал (Открытие → Работа → Закрытие)

> Три стадии, три масштаба. Пропуск Открытия = незапланированная работа.
> Пропуск Закрытия = незафиксированный результат.

| Масштаб | Открытие | Работа | Закрытие |
|---------|----------|--------|----------|
| **Сессия** | Определить задачу, прочитать контекст | Выполнение | Зафиксировать результат + capture |
| **День** | `bash ~/iwe-platform/scripts/day-open.sh` | Между Open и Close | `bash ~/iwe-platform/scripts/day-close.sh` |
| **Неделя** | Week Close (`week-close.sh`) | Работа по РП | **Strategy Session** → чтение протокола |
| **Месяц** | — | — | Month Close (первый Пн месяца, перед Strategy Session) |

## 3. Блокирующие правила

> **Важнее всех остальных.** Если правило ниже конфликтует с другими — побеждает оно.

### Правило 0. Недельный ритм (Rhythm Gate)

Система работает в еженедельном цикле:

| День | Событие |
|------|--------|
| **Вс/Пн утро** | Week Close (`week-close.sh`) — итоги, коммиты, создание нового WeekPlan |
| **Пн (strategy_day)** | **Strategy Session** — ревью НЭП, анализ недели, формирование WeekPlan |
| **Пн–Пт** | Day Open → Work → Day Close (ежедневный цикл) |
| **Первый Пн месяца** | Month Close (перед Strategy Session) |

**Pre-action Gate перед Strategy Session:**
- Убедиться, что выполнен Week Close прошлой недели
- Прочитать `memory/protocols/strategy-session.md`
- Прочитать `~/ds-strategy/docs/Session Agenda.md` (повестка)

День стратегической сессии задаётся в `memory/day-rhythm-config.yaml → strategy_day`.

### Правило 1. WP Gate — согласование работы

Любое **новое** задание, которого нет в текущем WeekPlan, требует явного согласования
перед началом работы.

**Ритуал согласования (обязательные шаги):**
1. **Verif Gate** — определить класс задачи (см. `memory/verification-classes.md`):
   trivial / closed-loop / open-loop / problem-framing
2. **Объявить:** Роль пользователя · Роль агента · Работа · РП · Класс верификации ·
   Метод · Оценка времени (~Nh) · Модель (текущая) — рекомендую [модель] ([причина])
3. **Дождаться явного «да»/«делаем»/«открывай».** Без согласия — не начинать.
4. Только после согласия — зарегистрировать РП в WP-REGISTRY, если задача перерастает
   в полноценный Work Package.

**Уровни автономности по классу:**
| Класс | Автономность | Captures | Модель |
|-------|-------------|----------|--------|
| trivial | Полная, без вопросов | Не нужны | flash |
| closed-loop | Автономная, с автоматической проверкой | По паттерну | flash/pro |
| open-loop | Совместная (предложить → утвердить) | Обязательны | pro |
| problem-framing | Экзоскелетная (вопросы → варианты → выбор) | Обязательны | pro |

**Исключения:**
- Исправление ошибки в уже открытой сессии
- Очевидный багфикс ≤5 мин
- Задача уже есть в текущем DayPlan или WeekPlan

### Правило 2. Close — фиксация результатов

При любом закрытии (сессии/дня/недели):
1. **Сначала commit + push** — проверить `git status --short` по ВСЕМ затронутым
   репозиториям. Незакоммиченные изменения → `git add -A && git commit` → push.
2. **Потом** — отчёт, capture, итоги.

Команды «заливай» / «запуши» / «закрывай» → commit + push без доп. вопросов.

### Правило 3. Pull-on-Touch

При первом **обращении** к репозиторию за сессию (любая операция — ls, read, find,
grep, edit, commit) — сделать `git pull --rebase`. Один раз на репозиторий, лениво.

Перед pull — проверить `git status`: dirty → stash или пропустить с пометкой
«вывод potentially stale». Rebase conflict → стоп + отчёт пользователю, без
автоматического разрешения. Нет сети → работать с локальной копией, помечать выводы
как potentially stale.

Применяется ко всем git-репозиториям в рабочем пространстве: `~/iwe-platform/`,
`~/ds-strategy/`, `~/ds-knowledge-index/`.

### Правило 4. Capture-to-Pack (захват знаний)

На каждом рубеже (закрытие сессии/дня/недели) агент проверяет: появилось ли знание,
которое стоит сохранить?

**Уровень автономности:** определяется в `params.yaml → capture_autonomy`
- `manual` — только по явной команде пользователя
- `propose` — агент предлагает, пользователь подтверждает (сейчас: `propose`)
- `auto` — агент пишет без подтверждения

**Типы capture:**
| Тип | Куда | Пример |
|-----|------|--------|
| **Правило** (1-3 строки) | `memory/distinctions.md` | «flash для рутины, pro для стратегии» |
| **Протокол/процесс** | `memory/protocols/` | «Как проводить стратегическую сессию» |
| **Доменное знание** | `~/ds-knowledge-index/captures/` | «Принципы работы Omniroute» |
| **Урок/рефлексия** | `~/ds-knowledge-index/drafts/` | «Не начинать сложную задачу без day-open» |

**Формат capture:**
```markdown
## YYYY-MM-DD: Тема

Что узнал: ...
Откуда: ...
Контекст: ...
```

Анонсировать действие: *«Capture: [что] → [куда]»*.

### Правило 5. Автономность (поведенческое)

**НЕ спрашивать разрешения** на очевидные действия. Задание → выполни → отчитайся.
Не заканчивать сообщение вопросом. Очевидные следствия (синхронизация, обновление
связанных файлов) — делать сразу. Факт, проверяемый самостоятельно (grep, БД,
конфиг), — проверять, а не спрашивать.

**Исключения (когда вопрос/согласование легитимны):**
1. **Необратимое разрушительное действие** (force push, удаление данных без бэкапа).
2. **WP Gate** (Правило 1) — создание нового РП требует согласования.
3. **Choice-question** — выбор между альтернативами («делаем X или Y?»).

Запрещены только yes/no-запросы согласия на готовое решение. Выбор между вариантами —
это коллаборация, нормально.

### Правило 6. Финиш > отлог (S-46)

Обнаружена дополнительная задача в текущей сессии (косяк, недоделка, следующий шаг) →
**дефолт = делаю сейчас**, НЕ «отдельный РП / техдолг» первым вариантом.

Choice-question «сейчас или потом?» = анти-паттерн (скрытое предложение отложить).

Исключения для отложения:
- Бюджет задачи вырос в ×2-×3 от оценки
- Требуется доменное решение или данные, которых нет
- Контекст полностью переключился (другая часть системы)

## 4. Pre-action Gates (проверки перед действием)

| Момент | Проверка |
|--------|----------|
| **Начало сессии** | Был ли Day Open сегодня? Если нет — предложить |
| **Новое задание** | **Verif Gate** — определить класс задачи (trivial/closed-loop/open-loop/problem-framing) |
| **Перед стратегической сессией** | Был ли Week Close? Есть ли WeekPlan draft? Выполнен ли Active WP Sweep? |
| **Первое обращение к репозиторию** | Pull-on-Touch (Правило 3) |
| **Перед закрытием** | Close (Правило 2): commit + push + capture |

Скилл — формализованный метод в виде скрипта. Вызов — через `bash ~/iwe-platform/scripts/skills/<skill-name>.sh`.

| Скилл | Команда | Зачем |

Каждый скилл вызывает сам себя через `startTask` с flash-моделью (кроме archgate — pro).

## 7. Правила коммуникации

- **Автономность:** не спрашивать разрешения на очевидные действия. Исключения — WP Gate,
  необратимые действия, choice-questions.
- **Ясность:** в начале ответа — краткий план (2-3 пункта). В конце — резюме сделанного.
- **Ответственность:** если что-то пошло не так — сказать сразу, предложить фикс.
- **Язык:** русский. Технические термины — на английском, если нет устоявшегося русского аналога.
- **Режим «на пальцах»:** если пользователь говорит «объясни», «на пальцах», «что сделали» —
  отвечать только по-русски, без имён файлов, путей, функций, коммитов. Через аналогии из
  физического мира.

## 8. Обновление FMT-шаблона

Автоматический механизм убран (WP-18, 2026-06-02). Причины:
- FMT не тегирует версии — единственный источник CHANGELOG.md на main
- `update.sh` перезаписывает адаптированные файлы, риск потери данных
- Каждая адаптация уникальна (Pi agent ≠ Claude)

**Ручная процедура:**
1. Открыть [CHANGELOG.md](https://github.com/TserenTserenov/FMT-exocortex-template/blob/main/CHANGELOG.md)
2. Сравнить с `fmt-sync-manifest.json` — какие файлы затронуты
3. Вручную применить изменения к адаптированным версиям
4. Обновить `fmt_version` в `params.yaml`
5. Закоммитить: `git add -A && git commit -m "fmt: update to vX.Y.Z"`
