# Distinctions — Ключевые различения

> Различения — это концептуальные границы, которые важны для принятия решений
> в IWE. Если различение не используется регулярно — оно не нужно.
> Добавляются только по мере возникновения в практике.

## Текущие различения

### Base vs DS vs Pack
- **Base** — принципы и форматы (что неизменно в рамках платформы).
- **DS** — производное: планы, код, решения (меняется каждую неделю).
- **Pack** — паспорт предметной области (появляется, когда знаний по теме
  накапливается достаточно для формализации).

### Открытие vs Работа vs Закрытие
- **Открытие** — планирование, согласование, загрузка контекста.
- **Работа** — исполнение.
- **Закрытие** — фиксация результата, capture, commit.

### Capture vs Draft
- **Capture** — готовое знание, можно сразу сохранить.
- **Draft** — сырая мысль, требует доработки.

### Flash (лёгкая модель) vs Pro (тяжёлая модель)
- **Flash** — быстрые задачи: рефакторинг, чтение, форматирование, рутина.
- **Pro** — стратегические сессии, архитектурные решения, сложный анализ.

### Sync по commit vs Sync по контексту
- **Sync по commit** — между вкладками/сессиями. Агент в новой вкладке не видит изменений из этой вкладки, пока они не закоммичены.
- **Sync по контексту** — внутри одной сессии. Агент помнит всё что обсуждали в этом чате.
- **Правило:** перед `startTask` или открытием новой вкладки — commit в affected репозиториях.

### Pull-on-read vs Pull-on-touch
- **Pull-on-touch** — первый pull при обращении к репозиторию за сессию. Работает для статичных файлов (код, конфиги).
- **Pull-on-read** — pull перед каждым чтением файлов в `ds-strategy/` и `ds-knowledge-index/` (DayPlan, WeekPlan, REGISTRY, Dissatisfactions, fleeting-notes, captures).
- **Правило:** перед чтением любого файла из этих репозиториев делать `git pull --rebase`. Не проверять «была ли другая сессия» — всегда. Если dirty — пропустить с пометкой «данные potentially stale».

### Close-обновление DayPlan
- **При закрытии сессии** (любой — вкладка, WP, день) — обновить DayPlan: отметить сделанное, добавить заметки, зафиксировать артефакты.
- **DayPlan — единый источник правды по статусу дня.** Без обновления DayPlan commit неполный.
- **Правило:** перед любым commit в ds-strategy проверить — нужно ли обновить DayPlan. Если сессия затронула WP, задачу или артефакт — DayPlan обновляется в том же commit.

### Экзоскелет vs Протез (HD #28)
- **Экзоскелет** — ИИ усиливает существующую функцию человека. Без него медленнее, но можно. Человек решает, ИИ предлагает.
- **Протез** — ИИ заменяет функцию. Без него нельзя. Человек делегирует мышление.
- **Тест:** завтра исчезнут ИИ-инструменты — продолжишь работать медленнее? Если да → экзоскелет.

### Элемент культуры vs Реализация элемента (HD #43)
- **Элемент культуры** — инвариант из DP.M.008: что и зачем (напр. «ОРЗ»). Меняется редко.
- **Реализация** — конкретный алгоритм в конкретной среде: как (напр. protocol-open.md). Меняется часто.
- **Следствие:** можно адаптировать реализацию под Pi agent + Aethon, не трогая инвариант.

### Pack-знание vs Реализационное решение (HD #29)
- **Pack-знание** — доменная истина: архитектура, паттерны, терминология эксперта. Source-of-truth.
- **Реализационное решение** — технический выбор: вендор, стек, конфигурация. Живёт в DS.
- **Правило:** при захвате знания сначала определить — это истина о домене или технический выбор?

### Note-Review vs Day Open
- **Note-Review** — вечерний классификатор: разбирает fleeting-notes по 7 категориям. Предлагает, не решает.
- **Day Open** — утренний планировщик: читает предложения Note-Review, формирует DayPlan. Принимает решения.
- **Note-Review без Day Open** = заметки разобраны, но не превращены в план.

### SSH к VPS — только read-only
- **Read-only** — сбор конфигов, логов, статусов. Всегда без запроса.
- **Изменяющие команды** — рестарт, запись, удаление, установка. Требуют явного разрешения.
- **Никаких исключений.** Нарушение подрывает доверие.
