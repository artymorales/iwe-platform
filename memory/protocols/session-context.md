# Протокол сохранения контекста между сессиями (Session Context)

> **Источник:** Н1 — «Знания и контекст теряются между сессиями — каждый раз с чистого листа»
> **Создан:** WP-26, 2026-06-27
> **Роль:** гарантировать, что агент при новом Open восстанавливает не только список задач,
>   но и ход мысли, открытые вопросы и неявные зависимости.
>
> **Принцип:** двухуровневый контекст — WP-уровень (что делаем) + сессионный уровень (о чём думаем).

---

## § Архитектура: два уровня контекста

| Уровень | Файл | Что хранит | Когда обновляется |
|---------|------|-----------|-------------------|
| **WP-уровень** | `~/ds-strategy/inbox/WP-N-context.md` | Артефакты, шаги, решения (work.md §2a) | На каждом рубеже (work.md §2a) |
| **Сессионный** | `~/ds-strategy/current/session-context.md` | Открытые вопросы, ход мысли, зависимости между РП | При КАЖДОМ Close (сессия/день) |

**Правило:** WP-уровень — для долгосрочного трекинга задачи. Сессионный — для быстрого восстановления
«на чём я остановился» при следующем Open. Оба обязательны.

---

## § Сессионный контекст (session-context.md)

### Формат

```markdown
# Контекст сессии — YYYY-MM-DD HH:MM

## Активные РП
- WP-N: что делали / на чём остановились
- WP-M: (ещё не начинали / ждёт X)

## Ход мысли
- Ключевой вывод / инсайт сессии (1 предложение)
- Текущая гипотеза (если есть)
- Что рассматривали, но отклонили

## Открытые вопросы
- вопрос 1
- вопрос 2

## Зависимости (cross-WP)
- WP-N ждёт решения из WP-M
- WP-K косвенно затронут изменениями в WP-N

## Первый шаг при следующем Open
- конкретное действие
- или «спросить пользователя о ...»
```

**Правила написания:**
- **Не копировать** WP-контекст (артефакты, список шагов) — это в WP-файле. Сессионный — только про мышление.
- **Один файл, перезапись.** Не дописывать, а заменять целиком при каждом Close. Хранит состояние последней сессии.
- **5 строк минимум, 20 максимум.** Если нечего писать — сессия была тривиальной (можно пропустить).
- **Timestamp в заголовке** — время Close, чтобы при Open было видно, насколько контекст свежий.

### Когда создаётся

Создаётся при Session Open (первое обращение к РП), если файла нет — агент пишет заглушку:
```markdown
# Контекст сессии — YYYY-MM-DD HH:MM

## Активные РП
- WP-N: начало работы

## Ход мысли
- (первая сессия)
```

### Когда обновляется

При **каждом Close** (сессия или день):
1. Записать ход мысли (даже если «не закончили, продолжаем»)
2. Актуализировать список активных РП
3. Обновить timestamp в заголовке

**Gate (БЛОКИРУЮЩЕЕ):** агент НЕ делает commit, пока `session-context.md` не обновлён.
Close Gate из close.md расширяется: commit только после session-context + WP-контекст.

## § Обновление WP-контекста

> **Канонический формат:** `work.md` §2a (содержит все поля: Открытые вопросы, Зависимости, статусы).
> Здесь — только правила, специфичные для сессионного уровня.

### Статусы — обязательный набор

Статус `**Статус:**` в заголовке WP-контекстного файла может быть только одним из:
- `in_progress` — активная работа
- `done` — завершён, ждёт архивации
- `blocked` — заблокирован внешней зависимостью

Никаких `closed`, `⏳`, `🔄` в заголовке контекстного файла. Эмодзи — только в WP-REGISTRY.

## § Inbox Cleanup

> **Полный протокол:** `close.md` §3a (Inbox Cleanup) + Week Close.
> Здесь — только дополнение для сессионного контекста.

При Week Close дополнительно:
1. Сверить WP-REGISTRY (все `in_progress`) с inbox
2. Если WP в `in_progress`, но контекстного файла нет → создать заглушку с `**Статус:** in_progress`
3. Вывести предупреждение: *«WP-N в работе, но контекстный файл отсутствовал. Создана заглушка.»*

## § Open Gate — гарантированная загрузка контекста

### При Session Open (дополнение к open.md §1b Handoff Gate)

После существующих проверок агент ОБЯЗАТЕЛЬНО:

1. **Прочитать `~/ds-strategy/current/session-context.md`:**
   - Есть файл → восстановить ход мысли, открытые вопросы, зависимости
   - Файла нет → первая сессия после внедрения протокола, создать заглушку
   - Timestamp старше 24ч → предупредить: *«Контекст устарел (N часов). Проверяю актуальность...»*

2. **Прочитать WP-контекст текущего РП** (уже есть в open.md — усилить):
   - Не только «прочитал», но и **сверить с реальностью**: все ли артефакты из `## Сделано` действительно существуют в репозитории?
   - Если расхождение → спросить пользователя

3. **Быстрая сверка активных РП:**
   - Какие РП сейчас `in_progress` по WP-REGISTRY?
   - Для каждого → существует ли контекстный файл?
   - Если нет → спросить: *«WP-N в работе, но контекстный файл отсутствует. Создать или этот РП уже не актуален?»*

## § Интеграция с Aethon memory (опционально, для P1 РП)

> **Aethon project memory** (`~/.aethon/memory/projects/.../MEMORY.md`) — persistent-канал,
> переживающий закрытие вкладок и перезапуски Pi agent. Используется как **резервный**
> канал для критического контекста.

**Когда использовать:**
- P1 РП в `in_progress` — записать ключевой контекст через `remember`
- Архитектурное решение, которое нельзя потерять
- Критическая зависимость между РП

**Формат записи в Aethon memory:**
```
[context] WP-N: [статус] — [последнее действие]. Следующий шаг: [X].
```

**Когда НЕ использовать:**
- P2/P3 РП (достаточно файлового контекста)
- Тривиальные решения
- Промежуточные состояния (хватит `session-context.md`)

**Правило:** Aethon memory — последний рубеж. Если файлы потеряны/испорчены, контекст
можно восстановить из Aethon memory. Но основной канал — файлы.

- [ ] WP-контекст прочитан и сверен с реальностью

- [ ] Контекст старше 24ч → предупреждение
