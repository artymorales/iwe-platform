# Протокол Закрытия (Close) — ОРЗ-фрактал

> **Три масштаба:** Сессия (Quick Close), День (Day Close), Неделя (Week Close).
> **Адаптация:** Pi agent + Aethon (нет `.claude/hooks/`, нет `TodoWrite`, нет `ScheduleWakeup`).
> **Источник:** AGENTS.md §2 (slim) → этот файл.

---

## § Различение: Сессия vs День vs Неделя

| | Quick Close (Сессия) | Day Close | Week Close |
|---|---|---|---|
| **Триггер** | «закрывай» / «всё» | «закрывай день» / `day-close.sh` | «закрывай неделю» / `week-close.sh` |
| **Бюджет** | ~3 мин | ~10-15 мин | ~20-30 мин |
| **Обязательно** | Всегда | Каждый день | Каждую неделю |
| **Что делает** | Commit + handoff | День целиком: метрики, WakaTime, FPF | Неделя: бэкап, validate, R-опросник |
| **Скрипт** | inline (этот протокол) | `bash ~/iwe-platform/scripts/day-close.sh` | `bash ~/iwe-platform/scripts/week-close.sh` |

> **«закрывай» без уточнения** → Quick Close (сессия) по умолчанию.

---

## § Quick Close (Сессия)

> **Роль агента:** кодировщик.
> **Бюджет:** ~3 мин. Только обязательные шаги.

### Шаг 1. Commit + Push (БЛОКИРУЮЩЕЕ)

**Проверить ВСЕ репозитории, которых касалась сессия:**
```bash
for repo in ~/iwe-platform ~/ds-strategy ~/ds-knowledge-index; do
  cd "$repo"
  if [ -n "$(git status --short)" ] || [ -n "$(git cherry -v)" ]; then
    git add -A
    git commit -m "сессия YYYY-MM-DD: краткое описание"
    git push
  fi
done
```

- **Только изменённые репозитории.** Не трогать нетронутые.
- **Commit message:** `сессия YYYY-MM-DD: [РП-N] [что сделано]`
- **Если dirty + не успел закоммитить** → stash и предупредить пользователя.

### Шаг 2. Capture — что узнали

Проверить: появились ли за сессию новые знания?

- **Правило** (1-3 строки) → `~/iwe-platform/memory/distinctions.md`
- **Доменное наблюдение** → `~/ds-knowledge-index/captures/`
- **Сырая мысль / урок** → `~/ds-knowledge-index/drafts/`
- **Нет нового знания** → пропустить молча

Анонс: *«Capture: [что] → [куда]»*.

**Проверить `capture_autonomy` в `params.yaml`:**
- `manual` — только по команде (пропустить, если не было команды)
- `propose` — предложить: *«Есть capture: [что] → [куда]. Записать?»*
- `auto` — записать без подтверждения

### Шаг 3. Обновить контекст РП (Handoff)

Обновить `~/ds-strategy/inbox/WP-N-context.md` (если файла нет — создать).

**Если задача НЕ завершена:**
- Обновить `**Последняя сессия:** YYYY-MM-DD HH:MM`
- Дополнить `## Сделано` (новые артефакты, если ещё не записаны)
- Проверить `## Осталось`: шаги актуальны? Добавить новые, если появились
- Записать решения сессии в `## Решения` (если ещё не там)
- Обновить `## Лог сессий`
- **Не менять `**Статус:**` — оставить `in_progress`**

**Если задача завершена:**
- `**Статус:** done`
- Все шаги в `## Осталось` → `- [x]`
- Обновить WP-REGISTRY: статус → `done`
- Переместить в архив:
  ```bash
  mv ~/ds-strategy/inbox/WP-N-context.md ~/ds-strategy/archive/wp-contexts/
  ```

**Формат контекстного файла — см. work.md §2a.**

### Шаг 4. Decision Capture — проверить

Решения пользователя уже должны быть записаны в `## Решения` контекстного файла
(work.md §3).

Проверить: есть ли незаписанные architectural/strategic решения за сессию?
Если да → добавить в контекстный файл.

### Шаг 5. Session Log

Сессия уже записана в `## Лог сессий` контекстного файла РП (work.md §2a).
Дополнительно — для сводного аудита:
```bash
echo "YYYY-MM-DD HH:MM–HH:MM | WP-N | done/in_progress | модель" >> ~/ds-strategy/inbox/session-log.md
```

### Шаг 5.5 Service Catalog — проверить

Если в сессии был «новый сервис / не связано с существующими» (work.md §0) —
проверить: нужно ли добавить строку в `~/iwe-platform/memory/service-catalog.md`?

Критерий: появился повторяемый процесс, протокол или подсистема, которой нет в каталоге.
Если да → добавить (новый S-номер).

---

### Шаг 5.6 Чеклист-верификация (опционально)

Если протокол содержит чеклист (T1-T17, Day Close, Week Close):

> *«Запустить верификацию чеклиста на flash? (да / нет)»*

При «да»: агент запускает sub-agent через `startTask` с flash-моделью.
Sub-agent получает: чеклист + факты (git status, log, captures → summary).
Задача sub-agent: проверить каждый пункт, найти пропуски.

**Почему sub-agent:** контекст исполнителя загрязнён уверенностью «я это делал»
→ независимый агент видит только чеклист и факты.

### Шаг 6. Отчёт пользователю

> **РП:** [WP-N] — [название]
> **Статус:** done / in_progress
> **Git:** [закоммичено ✅ / dirty ⚠️]
> **Capture:** [есть / нет / что]
> **Handoff:** [записан / done / не нужен]
> **Дальше:** [рекомендация]

## § Day Close

> **Триггер:** «закрывай день» / конец рабочего дня.
> **Скрипт:** `bash ~/iwe-platform/scripts/day-close.sh`

### Шаг 1. Проверить все открытые сессии
- Есть ли незакрытые сессии в `session-log.md`?
- Если да — выполнить Quick Close для каждой

### Шаг 2. WakaTime + метрики (если доступно)
- Проверить: есть ли WakaTime API key в окружении
- Если есть:
  ```bash
  wakatime-cli --today  # общее время за день
  ```
- Если нет — пропустить

### Шаг 3. Обновить DayPlan
- Отметить завершённые задачи
- Перенести незавершённое → завтра (или осознанно отложить)
- Сохранить в архив:
  ```bash
  mv ~/ds-strategy/current/DayPlan-YYYY-MM-DD.md ~/ds-strategy/archive/dayplans/
  ```

### Шаг 4. FPF — проверить актуальность
- Изменились ли фундаментальные принципы за день?
- Есть ли что добавить/уточнить в `AGENTS.md` или `memory/`?

### Шаг 5. Проверить t-checklist.md
Прочитать `~/iwe-platform/memory/t-checklist.md`.
Применить чеклисты, помеченные как **Day Close**.
- Если ни один чеклист не применим — пропустить.

### Шаг 6. Итог дня
Сказать пользователю:
> *«День закрыт. Сессий: N. Capture: N. РП: [статусы]. Завтра: [рекомендация].»*

## § Week Close

> **Триггер:** «закрывай неделю» / воскресенье / утро стратегической сессии.
> **Скрипт:** `bash ~/iwe-platform/scripts/week-close.sh`

### Шаг 1. Проверить Day Close за всю неделю
- Все ли дни недели закрыты?
- Если нет — выполнить Day Close для пропущенных

### Шаг 2. Бэкап
```bash
# Проверить, что все репозитории запушены
for repo in ~/iwe-platform ~/ds-strategy ~/ds-knowledge-index; do
  cd "$repo" && git status --short && git push
done
```

### Шаг 3. WeekPlan — итоги
- Отметить завершённые РП
- Перенести carry-over в следующую неделю
- Краткое резюме: что сделано / не сделано / почему

### Шаг 4. R-опросник (5-7 вопросов)
Прочитать `~/iwe-platform/memory/r-questionnaire.md` (если есть).
Ответить на вопросы → записать в Week Report.

Примерные вопросы:
1. Что было самым полезным на этой неделе?
2. Что не получилось и почему?
3. Какие знания нужно зафиксировать?
4. Что изменить на следующей неделе?

### Шаг 5. Week Report
Создать отчёт недели → `~/ds-strategy/archive/week-reports/Week-YYYY-WW.md`:
- Сделанные РП
- Capture summary
- R-ответы
- Carry-over
- Рекомендации на следующую неделю

### Шаг 6. Создать WeekPlan следующей недели
- Из carry-over
- Из стратегических целей
- Из inbox (если есть)

### Шаг 7. Итог недели
> *«Неделя [N] закрыта. РП завершено: N. Capture: N. Carry-over: N. Следующая неделя: [фокус].»*

---

## § Exit Protocol (при любом завершении)

| # | Шаг | Что делать |
|---|-----|-----------|
| 1 | **Артефакт** | Зафиксировать результат (коммит, файл, запись) |
| 2 | **Статус** | Обновить трекер (WP-REGISTRY, WP context) |
| 3 | **Handoff** | Если задача не завершена — записать «Осталось» для следующей сессии |
| 4 | **Уведомление** | Сообщить пользователю: что сделано, что дальше |

---

## § Исключения

- **Сессия ≤5 мин** (быстрый вопрос, без изменений) → Close не нужен
- **Несколько сессий подряд** (продолжение задачи) → Close только в конце
- **Прерванная сессия** (пользователь ушёл, таймаут) → handoff обязателен, даже если ничего не закончено
