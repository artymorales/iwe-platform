# Протокол Работы (Work) — ОРЗ-фрактал

> **Два масштаба:** День и Сессия.
> **Бюджет:** Day Work ~4-8h, Session Work ~30 мин–2h.
> **Адаптация:** Pi agent + Aethon (нет `.claude/hooks/`, нет hooks-событий, нет `ScheduleWakeup`).
> **Источник:** AGENTS.md §2 (slim) → этот файл.

---

## § Масштаб: День (Day Work)

> **Действует:** между Day Open и Day Close.
> **Роль агента:** стратег (ритм) + кодировщик (исполнение сессий).

### Правила дня

| # | Правило | Суть |
|---|---------|------|
| 1 | **Слот саморазвития первый** | Не переходить к текучке, пока слот не отработан или осознанно пропущен. **Блокирующее:** агент НЕ начинает операционные задачи, если слот не закрыт. Исключение: ТОЛЬКО ⚡ Срочное (конвейер остановлен) |
| 2 | **Каждая сессия = полный ОРЗ** | Open → Work → Close. Не пропускать |
| 3 | **Помидорки** | Личная практика пользователя (таймер). Агент не отслеживает — нет фонового таймера |
| 4 | **Напоминание о перерыве** | Если сессия >50 мин → сказать: *«Пора сделать перерыв. 5 минут.»* |
| 5 | **Сверка с планом** | Между сессиями: «Я ещё на плане дня?» Если нет — осознанное решение |
| 6 | **День стратегии** | Если сегодня `strategy_day` — не пропустить стратегическую сессию |

## § Масштаб: Сессия (Session Work)

> **Триггер:** После прохождения Open.
> **Цель:** выполнить задачу, сохранить фокус, зафиксировать знания на каждом рубеже.

---

### 0. Знай свои сервисы (БЛОКИРУЮЩЕЕ)

> **ОБЯЗАТЕЛЬНО перед началом работы.** Определи, какие сервисы IWE затронуты задачей.

1. Прочитай `~/iwe-platform/memory/service-catalog.md`.
2. Определи, какие сервисы (S01–S11) затрагивает задача:
   - Знаешь? → Назови их в плане: *«Работаю с S03 Session, S05 WP Registry»*
   - Не уверен? → **СТОП.** Сверься с каталогом. Предложи на согласование.
   - Задача не связана ни с одним сервисом (новая фича, исследование)? → Скажи: *«Новый сервис / не связано с существующими»*. Работай, но при Close проверь — нужно ли добавить строку в каталог.
3. Зачем:
   - Агент понимает, какую часть системы он меняет
   - Задача «поправить open.md» — это S03 (Session), не просто «отредактировать файл»
   - Если сервис затронут, но протокола нет → это сигнал к созданию

### 0.5. Триггеры посреди работы

Агент реагирует на эти команды в любой момент, не дожидаясь рубежа:

| Сообщение пользователя | Действие |
|------------------------|----------|
| `запиши идею: ...` / `на будущее: ...` / `fleeting: ...` | Записать в `~/ds-strategy/inbox/fleeting-notes.md`. Анонс: «Записал.» |
| `новый РП:` / `wp-new ...` | Создать РП через `wp-new.sh`. Записать в WP-REGISTRY, WeekPlan, контекстный файл. |
| `P0:` / `P1:` / `срочно:` | `strategy-protocol.md` §3: классифицировать, проверить инварианты, эскалировать. |
| `решение:` / `архитектурное:` | `strategy-protocol.md` §3: записать в `~/ds-strategy/decisions/`. |

После обработки триггера — вернуться к прерванной работе.

### 1. Начало работы: план ответа

В начале каждого ответа — краткий план (1-2 предложения):

> *«Сделаю [X] → затем [Y]. Если [проблема] — переключусь на [Z].»*

Если задача идёт не по плану — сказать сразу:
> *«Расхождение: [что пошло не так]. Меняю план: [новый подход].»*

### 2. Capture-to-Pack (на каждом рубеже)

> **Рубеж** = подзадача завершена, паттерн обнаружен, решение принято.

На каждом рубеже спросить: **есть ли знание для записи?**

| Тип знания | Куда | Когда |
|-----------|------|-------|
| **Правило** (1-3 строки, применимо везде) | `memory/distinctions.md` | Сразу |
| **Различение** (HD — hard distinction) | `memory/distinctions.md` | Сразу |
| **Доменное знание** (архитектура, паттерн, платформа) | `~/ds-knowledge-index/captures/` | На рубеже или Close |
| **Сырая мысль** (неоформленная, но ценная) | `~/ds-knowledge-index/drafts/` | Сразу |
| **Урок** («не делать X снова») | `~/ds-knowledge-index/captures/` | На рубеже |
| **Крупный урок** (заслуживает отдельного файла) | `memory/lessons/` (новый файл) | Close |
| **Протокол / процесс** (повторяемый алгоритм) | `memory/protocols/` | Close |
| **FAQ** (ответ на вопрос «как работает X?») | `~/ds-knowledge-index/captures/` → пометить `[FAQ]` | Сразу |
| **Зерно для поста** (связное рассуждение → публикация) | `~/ds-strategy/drafts/draft-list.md` (строка) | Close |

**Формат capture:**
```markdown
## YYYY-MM-DD: Тема

Что узнал: ...
Откуда: ...
Контекст: ...
```

**Анонс:** *«Capture: [что] → [куда]»*

**Уровень автономности** (из `params.yaml → capture_autonomy`):
- `manual` — ждать явной команды
- `propose` (сейчас) — агент предлагает, пользователь подтверждает
- `auto` — агент пишет без подтверждения

При `propose`: если пользователь не отвечает на анонс в течение 1-2 шагов — записать
без подтверждения (знание может потеряться).

**Capture-back:** после записи знания — вернуться к плану.
> *«Это знание что-то меняет в текущем плане или приоритетах?»*
>
> - Да → пересмотреть план (что теперь важнее?)
> - Нет → продолжаем

Цель: не дать новому знанию «уйти в архив» без влияния на текущую работу.

### Skill Discovery (на рубежах)

> Рядом с Capture-to-Pack. На каждом рубеже спросить: **есть ли повторяемое действие, которое стоит оформить как скилл?**

**Критерии кандидата в скилл:**
1. Действие повторяется ≥3 раз в разных сессиях
2. Имеет чёткий вход (аргументы) и выход (артефакт / решение)
3. Алгоритм устоялся (не меняется от сессии к сессии)

**Если все три → предложить:** *«Skill candidate: [название] — [что делает]. Оформить как `scripts/skills/[name].sh`?»*

**Куда:** `~/iwe-platform/scripts/skills/`

### Творческий конвейер (на рубежах)

> Рядом со Skill Discovery. На каждом рубеже спросить: **есть ли мысль, которая может стать постом / статьёй?**

Если да:
1. Записать зерно в `~/ds-strategy/drafts/draft-list.md` (одна строка: заголовок + концепт)
2. Не развивать сейчас — только зафиксировать. Развитие — отдельная сессия

**Цепочка:** сырая мысль → зерно в draft-list → сессия «развитие черновика» → пост

### 2a. Контрольная точка — обновление контекста РП

> На каждом значимом рубеже (завершена подзадача, создан артефакт, принято решение) —
> обновить WP context file. Защита от потери прогресса при обрыве сессии
> (закрытие вкладки, разрыв соединения).

**Формат контекстного файла:** `~/ds-strategy/inbox/WP-N-context.md`

```markdown
# WP-N: Название РП

**Статус:** in_progress | **Создан:** YYYY-MM-DD | **Обновлён:** YYYY-MM-DD HH:MM | **Последняя сессия:** YYYY-MM-DD HH:MM

## Сделано
- [x] Артефакт 1 (YYYY-MM-DD)
- [x] Артефакт 2 (YYYY-MM-DD)

## Осталось
- [ ] Конкретный следующий шаг
- [ ] Следующий за ним

## Решения
- YYYY-MM-DD HH:MM [approve]: Суть решения
- YYYY-MM-DD HH:MM [architectural]: Суть

## Блокеры
- (пусто или описание что мешает)

## Лог сессий
- YYYY-MM-DD HH:MM–HH:MM | модель | сделано / in_progress
```

**Что обновлять на рубеже:**
- Вычеркнуть выполненные шаги (`- [ ]` → `- [x]`) и добавить дату
- Добавить новые артефакты в `## Сделано`
- Записать решения пользователя в `## Решения`
- Обновить `**Последняя сессия:**` и `## Лог сессий`
- Если задача завершена полностью — `**Статус:** done`

**Не фанатично:** обновлять на значимых рубежах, не каждый коммит.

### 3. Decision Capture — фиксация решений пользователя

> **Зачем:** Через месяц не гадать «почему мы тогда выбрали этот подход?»
> **Пишутся ТОЛЬКО решения пользователя.** Решения агента в рамках своей работы — не писать.

**Тест:** «Если бы пользователь не сказал этого / не отклонил / не выбрал — выбор сделал бы сам агент?»
- Да → **НЕ писать** (это работа агента)
- Нет → **Писать** (это развилка, где выбор сделал пользователь)

**Что писать обязательно:**
- `architectural` — пользователь выбрал архитектуру / паттерн / технологию
- `strategic` — решение о приоритетах, scope, бюджете, направлении

**Что писать опционально** (если кажется важным для истории РП):
- `approve` — подтвердил нетривиальное предложение агента
- `reject` — отклонил предложение
- `redirect` — перенаправил задачу

**Что НЕ писать:**
- Агент выбрал реализацию/структуру/тест/имя файла
- Агент обнаружил и исправил баг
- Агент создал РП по своей инициативе
- Тривиальные «ок», «да», «норм», «делай»

**Куда:** `~/ds-strategy/inbox/WP-N-context.md` → `## Решения`.

**Стратегические решения (P0/P1 эскалация, смена приоритета):**
дополнительно → `~/ds-strategy/decisions/` по формату `strategy-protocol.md` §3.

**Формат строки:**
```markdown
- YYYY-MM-DD HH:MM [architectural]: Суть решения. Альтернативы: [что отклонили]
- YYYY-MM-DD HH:MM [strategic]: Приоритет X над Y. Причина: [контекст]
```

**Timestamp:** реальное время из сообщения пользователя или git-коммита. Не «утро/вечер».
**Сортировка:** последнее сверху (новая запись в начало секции).
**Bi-temporal:** каждое решение имеет `created` (дата строки) и `last_modified`. При пересмотре решения — не удалять строку, а добавить `(обновлено YYYY-MM-DD: [что изменилось])`. Это позволяет знать, когда решение последний раз пересматривалось.

### 4. Self-correction

Обнаружил расхождение с планом или прошлыми решениями?

1. **Остановиться.**
2. Сказать пользователю: *«Расхождение: [что не так]. [Где]. [Как исправить].»*
3. Не исправлять молча.
4. Если расхождение — в моём понимании задачи → запросить уточнение.

### 5. Pre-action Gates во время работы

| Момент | Проверка |
|--------|---------|
| **Перед `git commit`** | Если репо содержит AGENTS.md или протоколы — перечитать их (могли измениться) |
| **Перед предложением архитектурного решения** | Есть ли прецедент? Проверить memory/distinctions.md, captures |
| **Перед удалением/перемещением файла** | Есть ли ссылки на него? `grep -r "filename" ~/iwe-platform/memory/` |
| **После изменения протокола** | Обновить связанные ссылки в других файлах |

### 6. Коммуникация

| Ситуация | Действие |
|----------|----------|
| Начало ответа | Краткий план (1-2 предложения) |
| Шаг завершён | Что сделано + что дальше |
| Всё сделано | Итог: что получилось + что захвачено |
| Проблема | Сказать сразу: *«Проблема: [что]. Нужно: [что от пользователя].»* |
| Не хватает контекста | *«Нужен контекст: [что именно]. Подскажешь?»* |

**Не заканчивать сообщение вопросом.** Если нужен выбор — дать варианты.
Если нужна информация — сказать, что именно.

### 7. Верификация на рубеже (опционально)

Если на рубеже создан значимый артефакт:
> *«Артефакт [X] создан. Нужна верификация? (да / на Close / нет)»*

| Ответ | Действие |
|-------|----------|
| **Да** | Проверить сейчас (diff, grep, review) |
| **На Close** | Добавить в список Close (шаг 3c) |
| **Нет** | Пропустить (trivial / есть тесты) |

**Не задерживать работу.** Основная проверка — при Close.
