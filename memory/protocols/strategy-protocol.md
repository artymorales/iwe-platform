# Протокол работы со стратегией

> **Назначение:** стратегия живёт не только в сессию. Этот протокол описывает, как агент
> сверяется со стратегией каждый день, принимает решения mid-week и готовит данные к сессии.
>
> **Исполнитель:** Pi agent (один). При появлении мульти-агентности — вынести §§3-5
> в отдельные промпты роли «Стратег» (см. `~/ds-strategy/inbox/WP-27-context.md`).

---

## §1. Архитектура стратегии

├── decisions/                        ← Журнал стратегических решений (создаётся §3)

├── current/
│   ├── WeekPlan W{N}.md              ← План недели

├── memory/

│   │   ├── strategy-protocol.md      ← Этот файл — работа между сессиями
│   │   ├── open.md                   ← Day Open (обычный день)
│   │   ├── work.md                   ← Работа по РП

### Иерархия истины

```
Strategy.md (направления, бюджет)
  → Dissatisfactions.md (что не так → какие направления активировать)
    → WP-REGISTRY.md (конкретные РП)
      → WeekPlan (РП на эту неделю)
        → DayPlan (задачи на сегодня)
```

При противоречии — побеждает вышестоящий.

### Роли файлов

| Файл | Горизонт | Обновляется |
|------|----------|-------------|
| `Strategy.md` | Месяц–год | Стратегическая сессия |
| `Dissatisfactions.md` | Неделя–месяц | По появлению/снятию НЭП |
| `WP-REGISTRY.md` | Постоянно | При создании/закрытии РП |
| `WeekPlan` | Неделя | Week Close + стратегическая сессия |
| `DayPlan` | День | Day Open / Day Close |
| `decisions/` | Постоянно | §3 — при стратегическом решении |

## §2. Стратегический Day Open (session-prep)

> **Триггер:** `day-open.sh` определяет `strategy_day` (Пн).
> **Исполнитель:** Pi agent, сразу после Day Open, до стратегической сессии.
> **Цель:** подготовить черновик WeekPlan, чтобы сессия началась с готовых данных.

### Шаг 1. Итоги прошлой недели

Прочитать секцию «Итоги W{N-1}» в текущем WeekPlan (если есть — записана Week Review, §6).
Если секции нет — итоги не собирались, пометить: «⚠ Итоги прошлой недели отсутствуют.
Week Close не выполнен или §6 не отработал.»

### Шаг 2. Разбор inbox

Прочитать и классифицировать:

| Файл | Действие |
|------|----------|
| `fleeting-notes.md` | Каждую заметку: → в план / → capture / → на обсуждение / → удалить |
| `captures.md` | Записи с `[processed]` >14 дней → в архив. Без метки → оставить |
| `WP-*.md` (inbox) | Сверить статус с WP-REGISTRY. Done/merged/drop → в `archive/wp-contexts/` |

Сформировать блок повестки «📋 Inbox Triage»:
- ✅ Архив (отработано)
- 📌 Предложения в РП
- ❓ На решение (требует обсуждения с пилотом)

### Шаг 3. Проверка НЭП

Прочитать `Dissatisfactions.md`:
- Какие НЭП сняты за неделю?
- Какие новые появились?
- Есть ли НЭП без привязки к РП?

Сформировать блок повестки «НЭП» с предложениями.

### Шаг 4. Стратегическая сверка

Прочитать `Strategy.md` → фокус месяца, проекты, бюджет.
Сверить текущие РП из WP-REGISTRY с направлениями Strategy.md:
- РП без привязки к стратегии?
- Направление без РП?

Сформировать блок повестки «Стратегическая сверка» с расхождениями.

### Шаг 5. Active WP Sweep

```bash
bash ~/iwe-platform/scripts/active-wp-sweep.sh
```

Результат — в блок повестки как таблица активных РП.

### Шаг 6. Нерегулярные блоки

Прочитать `Session Agenda.md` → какие нерегулярные блоки применимы?
Если есть — добавить в повестку.

### Шаг 7. Формирование черновика WeekPlan

На основе шагов 1-6 сформировать `WeekPlan W{N}.md` со статусом `draft`:

```markdown
---
type: week-plan
week: W{N}
status: draft
---

# WeekPlan W{N}

## Итоги прошлой недели W{N-1}
[из шага 1]

## Повестка сессии
- [ ] Ревью прошлой недели
- [ ] 📋 Inbox Triage [из шага 2]
- [ ] НЭП [из шага 3]
- [ ] Стратегическая сверка [из шага 4]

## План на неделю (проект)

| # | РП | Бюджет | Статус | Проект |
|---|-----|--------|--------|--------|
| ... | ... | ... | pending | ... |

**Бюджет недели:** ~Nh
**ТОС недели:** [определить на сессии]
```

### Шаг 8. Сохранить и передать

1. Сохранить черновик в `~/ds-strategy/current/`
2. Закоммитить: `git add → commit → push`
3. Сообщить пользователю:
   > «Черновик WeekPlan готов. Итоги: N РП в плане, M НЭП активно, K заметок в inbox.
   > Начинаем стратегическую сессию?»

После утверждения пользователем — перейти к `strategy-session.md`.

## §3. Принятие стратегических решений вне сессии

> **Триггер:** появление нового сигнала mid-week (НЭП, инцидент, идея, запрос),
> который может изменить приоритеты текущего WeekPlan.
> **Исполнитель:** Pi agent.

### Классификация сигнала

| Уровень | Смысл | Пример | Действие |
|---------|-------|--------|----------|
| **P0** | Критический | Прод упал, потеря данных, блокирует других агентов | В план **немедленно**. Уведомить пилота. ТОС — только по решению пилота |
| **P1** | Высокий | Новая сильная НЭП, угроза дедлайну | В план **в течение дня**. Может вытеснить P2/P3 |
| **P2** | Средний | Идея, улучшение, не срочно | **Отложить** до следующего Day Plan |
| **P3** | Низкий | «Было бы неплохо когда-нибудь» | В **backlog** с триггером активации. Без явного решения — не в WeekPlan |

### Инварианты (не нарушать без пилота)

1. **ТОС** (главный приоритет недели) — не меняется автоматически. Даже при P1 — запросить решение пилота
2. **Red Lines** из `Strategy.md` (§Red Lines) — не пересекать:
   - Бюджет недели: сумма часов ≤ физический лимит
   - Runway: финансовая подушка ≥ N недель
   - Дедлайны: критический перенос — только через решение пилота
3. **Бюджет** — нельзя суммарно запланировать > физического лимита часов в неделю

### Алгоритм при P0/P1

1. Классифицировать сигнал
2. Проверить инварианты: что нарушается?
3. Если инварианты не задеты → добавить РП в WeekPlan, обновить бюджет
4. Если инварианты задеты → сформировать choice-question пилоту:
   ```
   Сигнал: <описание> (P0/P1)
   Конфликт: <что нарушается — ТОС / бюджет / Red Line>
   Варианты:
     А. Принять, вытеснив <РП> (бюджет +Nh)
     Б. Отложить до <дата / следующей сессии>
     В. Разбить на меньшие задачи
   ```
5. Без ответа пилота — НЕ менять WeekPlan

### Журнал решений (decisions/)

При ЛЮБОМ стратегическом решении (P0/P1 эскалация, смена приоритета, архитектурный выбор) —
создать файл в `~/ds-strategy/decisions/`:

```markdown
# Решение: <тема>
**Дата:** YYYY-MM-DD
**Контекст:** <что произошло, почему потребовалось решение>
**Варианты:**
- А: ... (последствия)
- Б: ... (последствия)
**Решение:** выбрали X, потому что ...
**Кто принял:** пилот / агент по классу автономности
**Связанные РП:** WP-NNN, WP-MMM
```

Файл: `~/ds-strategy/decisions/YYYY-MM-DD-<краткий- slug>.md`

## §4. Ежедневная сверка со стратегией

### В WP Gate (новое задание)

При WP Gate (Правило 1 AGENTS.md), после Verif Gate, добавить вопрос:

> «Эта задача ведёт к какому результату месяца из Strategy.md?»

Если ответа нет — задача либо отклоняется, либо помечается как «вне стратегии»
с осознанным решением пилота.

Классы автономности при сверке:
- **trivial / closed-loop** — агент сам проверяет alignment, при расхождении докладывает
- **open-loop / problem-framing** — обязательный вопрос пилоту

### В Day Open (обычный день)

После формирования DayPlan проверить:
- Все задачи дня привязаны к РП из WeekPlan?
- WeekPlan всё ещё соответствует фокусу месяца из Strategy.md?

Если расхождение — пометить в DayPlan:
> ⚠ Стратегическая сверка: [конкретное расхождение]

### В Day Close

При подведении итогов дня добавить:
- **Стратегический инсайт дня** (если был): одно предложение. Что узнали о стратегии сегодня?
- Записать в `DayPlan → Итог дня → Captures` или в `fleeting-notes.md`

## §5. Стратегический контекст для агентов

При запуске `startTask` для подчинённого агента — включить:

1. **Фокус месяца** (из Strategy.md → Фокус месяца) — 1-2 предложения
2. **ТОС недели** (из WeekPlan) — что самое важное
3. **Целевой РП** — привязка задачи к конкретному РП
4. **Класс верификации** — чтоб агент знал уровень автономности

Шаблон bridgePrompt:

```
[Стратегический контекст]
Фокус месяца: <из Strategy.md>
ТОС недели: <из WeekPlan>
Целевой РП: WP-NNN — <название>
Класс: <trivial/closed-loop/open-loop/problem-framing>

[Задача]
<промпт задачи>
```

## §6. Week Review (итоги недели)

> **Триггер:** `week-close.sh` (Вс вечер / Пн утро перед сессией).
> **Исполнитель:** Pi agent, после коммитов week-close.
> **Цель:** записать фактическую секцию «Итоги W{N}» в WeekPlan — вход для §2 (session-prep).

### Шаг 1. Сбор коммитов

По каждому репозиторию (`iwe-platform`, `ds-strategy`, `ds-knowledge-index`):

```bash
git -C <repo> log --since="last monday 00:00" --until="today 00:00" --oneline --no-merges
```

Сопоставить коммиты с РП из текущего WeekPlan.

### Шаг 2. Статистика

- Completion rate: X/Y РП (N%)
- Коммитов всего
- Активных дней (дни с коммитами)

### Шаг 3. Статусы РП

Пройти каждый РП из WeekPlan, определить: done / partial / not started.
Для каждого — короткий комментарий (1 предложение).

### Шаг 4. Калибр (план/факт)

Для каждого РП из WeekPlan:

| # | РП | План (h) | Факт (h) | Отклонение |
|---|-----|----------|----------|------------|
| ... | ... | X | Y | 🟢 ≤20% / 🟡 20-50% / 🔴 >50% |

Факт — из Day Close или оценка агента.

### Шаг 5. Инсайты и carry-over

- Что получилось хорошо
- Что можно улучшить
- Блокеры
- Carry-over на следующую неделю

### Шаг 6. Запись в WeekPlan

Добавить (или обновить) секцию в текущем WeekPlan:

```markdown
## Итоги W{N}

### Метрики
- **РП:** X/Y завершено (N%)
- **Коммитов:** N в M репо
- **Активных дней:** N/7

### РП

| # | РП | Статус | Комментарий |
|---|-----|--------|-------------|
| ... | ... | done/partial/⬜ | ... |

### Калибр

| # | РП | План (h) | Факт (h) | Отклонение |
|---|-----|----------|----------|------------|
| ... | ... | X | Y | 🟢/🟡/🔴 |

### Инсайты
- ...

### Carry-over
- ...

### Контент-план W{N+1} (опционально)
- ...
```

### Шаг 7. Зафиксировать

```bash
cd ~/ds-strategy && git add current/ && git commit -m "week-review: итоги W{N}" && git push
```
