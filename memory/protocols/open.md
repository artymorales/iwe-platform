# Протокол Открытия (Open) — ОРЗ-фрактал

> **Два масштаба:** День и Сессия. Триггер определяет масштаб.
> **Адаптация:** Pi agent + Aethon (нет `.claude/hooks/`, `TodoWrite`, `ScheduleWakeup`).
> **Источник:** AGENTS.md §2 (slim) → этот файл.

---

## § Различение: День vs Сессия

| | Day Open | Session Open |
|---|---|---|
| **Триггер** | Первая сессия дня (утро) | Любое новое задание |
| **Что делает** | Запускает ритм дня, проверяет обновления, планирует | Определяет задачу, контекст, согласование |
| **Скрипт** | `bash ~/iwe-platform/scripts/day-open.sh` | Inline (этот протокол) |
| **Повтор** | Один раз в день | Каждый раз при новом задании |

**Если день ещё не открыт** (проверить: есть ли свежий DayPlan в `~/ds-strategy/current/`) — предложить сначала Day Open.

## § Day Open — Открытие дня

> **Триггер:** Первое сообщение дня / команда «открывай день».
> **Бюджет:** ~5-10 мин.
> **Скрипт:** `bash ~/iwe-platform/scripts/day-open.sh`

### Шаг 1. Проверка обновлений FMT
Если `params.yaml → fmt_check_on_open: true`:
```bash
bash ~/iwe-platform/scripts/fmt-version-check.sh --notify
```
- Exit 0 → версия актуальна, продолжать
- Exit 1 → есть новая версия. Сказать:
  *«📦 Доступна новая версия FMT vX.Y.Z. Запустить просмотр изменений?»*
- При согласии → `bash ~/iwe-platform/scripts/fmt-diff.sh`
- Отчёт категоризировать (🔵 NEW / 🟡 CHANGED / 🔴 DEPRECATED)

### Шаг 2. Прочитать конфигурацию дня
```yaml
# ~/iwe-platform/memory/day-rhythm-config.yaml
strategy_day: monday
pomodoro:
  work_minutes: 25
  break_minutes: 5
```

**Стратегический день?** Если сегодня == `strategy_day`:
- Убедиться, что Week Close прошлой недели выполнен
- Прочитать `memory/protocols/strategy-session.md`
- Прочитать `~/ds-strategy/docs/Session Agenda.md`

### Шаг 3. Прочитать планы
- Последний DayPlan (если есть) → `~/ds-strategy/current/`
- WeekPlan → `~/ds-strategy/current/`
- WP-REGISTRY → `~/ds-strategy/docs/WP-REGISTRY.md`

### Шаг 4. Запустить day-open.sh
```bash
bash ~/iwe-platform/scripts/day-open.sh
```
Скрипт создаёт/обновляет DayPlan. После — сказать:
> *«День открыт. DayPlan на сегодня — [кратко]. Готов к работе.»*

## § Session Open — Открытие сессии

> **Триггер:** Любое новое задание (кроме Day Open / Day Close / Week Close).
> **Применять ПОЛНОСТЬЮ, включая WP Gate — блокирующая проверка.**

### Шаг 0. Загрузка контекста (первая сессия дня / новый контекст)

Если это первая сессия дня ИЛИ контекст переключился на другой проект:
1. Прочитать `~/iwe-platform/AGENTS.md` (особенно §1 Архитектура, §2 ОРЗ, §3 Блокирующие правила)
2. Прочитать `~/iwe-platform/memory/distinctions.md`
3. Прочитать `~/iwe-platform/memory/verification-classes.md`
4. Прочитать `~/iwe-platform/params.yaml` (пути, модели, настройки)
5. Статус FMT: `grep fmt_version ~/iwe-platform/params.yaml`

**Не перечитывать,** если контекст не переключался (остались в том же проекте/задаче).

---

### Шаг 1. WP Gate — блокирующая проверка

**ДО ЛЮБОГО ДЕЙСТВИЯ** над задачей.

#### 1a. Задача есть в плане?
1. Прочитать WeekPlan → `~/ds-strategy/current/`
2. Задача совпадает с пунктом WeekPlan? → Да: перейти к шагу 1b. Нет: шаг 1c.
3. Не нашёл в WeekPlan → grep WP-REGISTRY.md (полный список):
   ```bash
   grep -n "WP-" ~/ds-strategy/docs/WP-REGISTRY.md | grep -i "задача\|тема\|тег"
   ```
   Нашёл → шаг 1b. Не нашёл → шаг 1c.

#### 1b. Задача в плане — работаем
- Ссылаемся на номер РП.
- **DayPlan Gate:** если задача не в DayPlan → добавить строку (если сегодня не strategy_day, там DayPlan не обязателен).
- **Sync Gate:** прочитать контекст РП (если есть context file в `~/ds-strategy/inbox/`) → синхронизировать понимание: что уже сделано, что осталось.
  - Если state-файл свежий (создан сегодня) → пропустить sync (уже актуально).
  - Если stale или нет → прочитать WP context file заново.
- Перейти к шагу 2 (Ритуал).

#### 1c. Задачи нет в плане — СТОП
1. Сказать: *«Этой задачи нет в плане на неделю.»*
2. Показать текущий WeekPlan (таблица РП):
   ```
   WP-N | Название | Статус | Оценка
   ```
3. Спросить: нужно ли создать новый РП или это исключение?
   - **Новый РП:** артефакт? формулировка? какой репозиторий? бюджет?
   - **Исключение** (≤15 мин, очевидный багфикс, вопрос без изменений файлов): описать причину, зафиксировать в open-sessions.log.
4. Если новый РП → записать в 4 места:
   1. `WP-REGISTRY.md` — новая строка
   2. `WeekPlan` — новый пункт
   3. WP context file → `~/ds-strategy/inbox/WP-N-teaser.md` (краткое описание)
   4. open-sessions.log (шаг 4)
   Нумерация: последовательные целые (следующий за max в WP-REGISTRY).

**Второе исключение подряд** → *«Вторая задача вне плана. Перерастает в РП — записать?»*

---

### Шаг 2. Ритуал согласования

#### 2a. Определить класс верификации (Verif Gate)
По `memory/verification-classes.md`. Три вопроса:

1. **Результат очевиден?** → Да: **trivial**. Нет: ↓
2. **Есть дешёвая автоматическая проверка?** (тест, линтер, diff, grep) → Да: **closed-loop**. Нет: ↓
3. **Решение известно?** → Да: **open-loop**. Нет: **problem-framing**.

#### 2b. Объявить ритуал
> **Роль пользователя:** [что делает пользователь]
> **Роль агента:** [стратег / кодировщик / экстрактор]
> **Работа:** [что делаем]
> **РП:** [WP-N, если есть]
> **Класс верификации:** [trivial / closed-loop / open-loop / problem-framing]
> **Метод:** [как делаем]
> **Оценка:** ~Xh
> **Модель:** [flash / pro] — рекомендую [модель] ([причина])

**Пример:**
> **Роль пользователя:** разработчик платформы
> **Роль агента:** кодировщик
> **Работа:** расширить протокол Open проверкой обновлений FMT
> **РП:** новая задача (предложу завести WP-77)
> **Класс верификации:** closed-loop (diff покажет, что изменения применились)
> **Метод:** отредактировать memory/protocols/open.md, проверить `git diff`
> **Оценка:** ~1h
> **Модель:** flash — рекомендую flash (closed-loop, без captures)

#### 2c. Экзоскелетный режим (problem-framing)
Если класс **problem-framing** — НЕ предлагать решение сразу. Вместо этого:
1. Задать 3 вопроса: **Что?** (конечная цель), **Зачем?** (критерий успеха), **Ограничения?** (рамки, ресурсы)
2. Предложить 2-3 варианта подхода с trade-offs
3. Дождаться выбора пользователя
4. Только после выбора — начинать работу

#### 2d. Дождаться согласования
Явное «да», «делаем», «начинай». Без согласия — не начинать.

Для **trivial / closed-loop** в уже открытой сессии (пользователь уже дал контекст) — можно пропустить шаг 2b-2d, достаточно быстрой справки.

### Шаг 3. Pull-on-Touch

Перед первым обращением к любому репозиторию за сессию:
```bash
git -C ~/iwe-platform pull --rebase
```
- Dirty → `git stash` или пропустить с пометкой «вывод potentially stale»
- Rebase conflict → СТОП, отчёт пользователю, без автоматического разрешения
- Нет сети → работать с локальной копией, помечать как potentially stale
- Делать один раз на репозиторий (для `~/iwe-platform/`, `~/ds-strategy/`, `~/ds-knowledge-index/`)

---

### Шаг 4. Регистрация в open-sessions.log

Для задач не-exception (кроме trivial-поправок внутри сессии):
```bash
echo "YYYY-MM-DD HH:MM | WP-N | модель | описание" >> ~/ds-strategy/inbox/open-sessions.log
```

**Формат:** одна строка на сессию. Дата и время начала, РП (если есть), модель, краткое описание.

---

### Шаг 5. Артефактор (для больших задач)

Если класс ∈ {open-loop, problem-framing} ИЛИ оценка ≥3h:
1. Создать **этапную карту** в WP context file (секция `## Этапы` в конце):
   ```markdown
   ## Этапы
   1. [что делать]
   2. [что зачем]
   3. ...
   ```
2. Каждый этап — отдельная веха с критерием готовности.

Если trivial/closed-loop И оценка <3h → пропустить.

---

### Шаг 6. Начать выполнение

Первым делом — создать **промежуточный артефакт**, чтобы зафиксировать направление:
- Для документации: заголовок + bullets
- Для кода: структура файла
- Для исследования: план поиска

## § Исключения (WP Gate не нужен)

| Тип | Пример |
|-----|--------|
| ≤15 мин | Переименовать переменную, поправить опечатку |
| Вопрос без изменений файлов | «Что делает этот протокол?» |
| Очевидный багфикс | Исправление сломанной ссылки |
| Продолжение текущей задачи | Следующий шаг в уже открытой сессии |

**Второе исключение подряд** → предупредить и предложить создать РП.

---

## § Общие правила при Open

- **Session Open ≠ Day Open.** Если Day Open был ранее — Session Open не повторяет контекст.
- **Продолжение задачи** (вернулись к тому же РП) — повторный Ритуал не нужен.
- **Смена модели внутри сессии:** делегировать sub-task через `startTask` (Aethon) только **вниз** (pro → flash). Переключение вверх — только через `/model` (пользователь вручную).
