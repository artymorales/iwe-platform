# Классы верификации (Verif Gate)

> Определяет режим работы агента и модель. Применяется при **каждом новом** задании
> как часть WP Gate (Правило 1).

## 4 класса

| Класс | Проверка результата | Режим | Модель | Captures |
|-------|-------------------|-------|--------|----------|
| **trivial** | Не нужна (результат очевиден) | Агент автономно, без captures | flash | не нужны |
| **closed-loop** | Дешёвая, автоматическая (тесты, linter, diff) | Агент автономно | flash/pro | не обязательны* |
| **open-loop** | Дорогая, отложенная (ревью, approval) | Совместно, captures обязательны | pro | обязательны |
| **problem-framing** | Неизвестная (решение неизвестно) | Экзоскелетный: вопросы > ответы | pro | обязательны |

\* closed-loop может иметь captures только если обнаружен новый паттерн.

## Как определить класс

**Задай 3 вопроса:**

1. **Результат очевиден?** Я уверен, что получу правильный ответ без проверки?  
   → Да: **trivial**. Нет: иди к 2.

2. **Есть дешёвый автомат проверки?** Тест, линтер, валидация схемы, grep, diff?  
   → Да: **closed-loop**. Нет: иди к 3.

3. **Решение известно?** Я знаю, ЧТО должно получиться, но проверить можно только человеком?  
   → Да: **open-loop**. Нет (неизвестно): **problem-framing**.

## Режимы работы

### trivial — автономный (flash)

- flash (дешёвая модель)
- Не спрашивать, делать сразу
- Без captures (результат самоочевиден)
- Примеры: `git mv`, rename variable, форматирование Markdown, поиск grep

### closed-loop — автономный с верификацией (flash/pro)

- flash (или pro, если задача сложная)
- Проверить автоматически (тест → pass/fail, diff → чистый/нет)
- Captures только если открылся новый паттерн
- Примеры: рефакторинг с тестами, код по спецификации, линтинг

### open-loop — совместный (pro)

- pro (тяжёлая модель) — обязательна
- Совместная работа: агент предлагает → человек подтверждает
- Captures обязательны на Close (что узнал, куда записал)
- Примеры: архитектурное решение, новый протокол, ревью стратегии

### problem-framing — экзоскелетный (pro)

- pro — обязательна
- **Агент НЕ предлагает решение сразу.**
- Вместо этого: 3 уточняющих вопроса (Что? Зачем? Ограничения?)
- Ответы → 2-3 варианта подхода с trade-offs → пользователь выбирает → работа
- Captures обязательны
- Примеры: «как построить систему X?», «какую архитектуру выбрать?»

## Переключение модели

| Сценарий | Что делает агент |
|----------|-----------------|
| **Задача проще текущей модели** | *«Эта задача trivial, рекомендую переключиться на flash.»* — предлагает, пользователь переключает сам |
| **Задача сложнее текущей модели** | *«Эта задача open-loop, рекомендую pro.»* — предлагает, пользователь переключает сам |
| **Delegation** (sub-task внутри сессии) | Delegation ТОЛЬКО вниз: pro→flash, pro→pro. Вверх — только через /model |

## Ритуал объявления

При открытии сессии агент объявляет:

> **Роль пользователя:** [разработчик / пользователь / ...]  
> **Роль агента:** [стратег / экстрактор / кодер / ...]  
> **Работа:** [что делаем]  
> **РП:** [WP-N, если есть]  
> **Класс верификации:** [trivial / closed-loop / open-loop / problem-framing]  
> **Метод:** [как делаем]  
> **Оценка:** ~Xh  
> **Модель:** [текущая] — рекомендую [модель] ([причина])
